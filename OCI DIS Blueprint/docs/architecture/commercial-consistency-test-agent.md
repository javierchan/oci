# Commercial Consistency Test Agent

## Purpose

The Commercial Consistency Test Agent is an independent engineering control used
while implementing or changing OCI pricing behavior. It is separate from the App's
`Official Source Governance Agent`: it does not appear in Agent Operations, run in
the production agent worker, or read/write customer data.

Its job is to challenge the implementation against persisted official documentary
evidence and independently asserted quotation fixtures before the primary engineer
can treat a commercial change as validated.

## Inputs

- the exact implementation diff and affected database contract;
- approved Oracle Price List, Supplement, public price, products, metrics, and
  presets evidence already available in this workspace;
- field-level source authority and the candidate/release audit trail;
- deterministic formula, boundary, tier, rounding, proration, dependency, and
  publication fixtures;
- API, export, UI, and BOM provenance contracts affected by the change.

## Required Review

1. Reconcile every changed field with its authoritative source.
2. Verify that source states such as blank, not offered, Always Free, conditional,
   and numeric zero remain distinct.
3. Validate formula semantics with independent expected values, including zero,
   below-minimum, boundary, above-boundary, fractional, and environment cases.
4. Confirm that generated candidates never self-approve and that failed or stale
   fixtures block the affected release scope.
5. Trace approved data from source artifact to normalized records, rule family,
   mapping, release, BOM line, monthly periods, export, agent evidence, and UI.
6. Verify that partial releases expose included and excluded SKUs and never present
   scoped zero exceptions as complete catalog coverage.
7. Record every unresolved discrepancy in
   `docs/audits/m51-commercial-catalog-deviations.md` with evidence and severity.

## Responsibility Escalation

When the independent review finds a recurring, machine-observable condition, the
primary engineer must convert it into a deterministic rule or regression fixture.
If the condition is also useful during ongoing operations, the App's Official
Source Governance Agent may receive a new **read-only reporting responsibility**.
The App agent must never receive approval, pricing calculation, mapping mutation,
exception disposition, release promotion, or BOM mutation authority.

Current operational responsibilities promoted from this control are:

- report whether the approved commercial release is complete or partial;
- list excluded App mapping SKUs and persisted exclusion reasons;
- identify candidates generated by an outdated factory version;
- identify missing, outdated, pending, or failed quotation rule fixtures;
- direct an administrator to deterministic revalidation without claiming approval.

## Completion Evidence

The agent review is complete only when its findings are reconciled into code,
tests, or the deviation audit; all Python tests run from the workspace `.venv`;
frontend, OpenAPI, Docker, browser, and visual gates relevant to the change pass;
and the main engineer records remaining risks without overstating coverage.
