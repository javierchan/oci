## M1 - Schema Readiness Guardrails for Admin Synthetic Routes

**Tier:** High
**Status:** open

### Problem

The Admin Synthetic Lab can hard-fail in a live environment when the code and database schema drift out of sync. In this run, the page rendered a generic `Failed to fetch` banner because the new `synthetic_generation_jobs` table had not been migrated into the running database yet.

### Root Cause

The new routes and services correctly depend on the new table, but the runtime has no explicit readiness guard that tells operators or the UI that a required Alembic migration is still pending.

### Target State

- Schema drift is detected before a user hits a broken admin page.
- The API exposes an actionable readiness failure instead of a raw 500-backed browser fetch error.

### Files Most Likely To Change

- `apps/api/app/main.py` - health/readiness signaling
- `apps/api/app/routers/admin_synthetic.py` - route-level handling if readiness is not met
- `apps/api/app/services/synthetic_service.py` - defensive error mapping if the schema is unavailable
- `README.md` or `docs/progress.md` - operator/dev runbook note

### Action Plan

1. Decide the readiness contract for pending migrations: fail health/readiness, expose migration status, or return a structured admin-surface error.
2. Implement pending-migration detection so schema-sensitive routes stop failing as opaque 500s.
3. Ensure the resulting error still returns browser-readable JSON/CORS behavior for localhost admin usage.
4. Document the expected migration step for local and deployment workflows.

### Exit Criteria

- A stale schema no longer produces a generic `Failed to fetch` symptom on `/admin/synthetic`.
- `curl` against the affected route returns a structured, actionable failure when migrations are pending.
- Backend tests and the main quality gates remain green.
