## M1 — Persist Synthetic Generation Jobs

**Tier:** High
**Status:** deferred

### Problem

Synthetic generation is currently executable and validated, but it is initiated through a reusable service plus a script entrypoint rather than through a first-class persisted job model. That limits operational traceability, replay governance, and admin-only lifecycle control.

### Root Cause

`M24` deliberately shipped the smallest safe slice that delivered a real governed synthetic project without prematurely implementing the entire Admin Synthetic Lab module. The documented future model was intentionally left for a follow-on phase.

### Target State

- Synthetic generation requests are stored as durable records with input parameters, status, artifacts, validation results, and actor metadata.
- Synthetic jobs can be queried, retried, and audited through supported product routes instead of script-only execution paths.

### Files Most Likely To Change

- `apps/api/app/models/` — add persisted synthetic job model
- `apps/api/migrations/versions/` — add Alembic migration
- `apps/api/app/schemas/` — add request/response DTOs
- `apps/api/app/services/synthetic_service.py` — split orchestration from job persistence
- `docs/architecture/admin-synthetic-lab.md` — align documentation with implemented state

### Action Plan

1. Add a `SyntheticGenerationJob` data model plus migration that captures governed inputs, status, artifacts, and validation outcomes.
2. Refactor `synthetic_service.py` so deterministic dataset construction remains reusable while execution state transitions are persisted around it.
3. Add schemas and service methods for create/get/list/retry behavior with service-layer authorization.
4. Validate the model and migration with focused backend tests plus live DB migration/seed compatibility checks.

### Exit Criteria

- A persisted synthetic-job table exists and migrates cleanly from the current head.
- Synthetic generation can be submitted and queried through supported backend contracts.
- Job records retain artifact references and deterministic validation outcomes.
- Relevant backend tests and existing quality gates remain green.

## M2 — Add Admin Synthetic Lab Submission and Observability Surfaces

**Tier:** Medium
**Status:** deferred

### Problem

The current synthetic capability is discoverable in generated project artifacts and project metadata, but there is no complete admin submission and observability UI for operators to initiate, monitor, or manage synthetic-generation jobs inside the product.
